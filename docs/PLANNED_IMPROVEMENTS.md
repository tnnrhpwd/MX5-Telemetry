## 1. Dedicated 12V Power Circuit

### The Problem

Currently the system is powered from OBD-II Pin 16 (12V) through a single LM2596 buck converter to a breadboard. This has several weaknesses:

- **Breadboard** is not vibration-resistant (contacts can loosen while driving)
- **Single point of failure** — one buck converter feeds everything
- **No ignition-switched power control** — system is always on when OBD provides 12V
- **No graceful shutdown** — Pi can corrupt its SD card if power is suddenly cut
- **LM2596 efficiency** — these modules are adequate but not optimal for automotive noise environments

### Option A: Automotive USB-C PD Power Supply (Simplest)

Use a purpose-built automotive 12V-to-USB-C power delivery module. These are designed for automotive voltage ranges (9-36V), include EMI filtering, and output regulated 5V.

| Part | Description | Approx. Price | Source |
|------|-------------|---------------|--------|
| **DFRobot DFR0756** | 9-36V to 5V/5A USB-C, automotive grade | $12-15 | DFRobot, Amazon |
| **Tobsun EA15-5** | 12V to 5V 3A, encapsulated DC-DC | $8-12 | Amazon, eBay |
| **DROK 12V to 5V 5A** | Buck converter with USB-C output | $8-10 | Amazon |

**Pros:** Plug-and-play, proper filtering, designed for automotive voltage swings (engine cranking dips to ~9V, load dumps to ~16V)
**Cons:** Still need separate power for Arduino/MCP2515 modules

### Option B: Dual Buck Converter Setup (Recommended)

Use two separate buck converters — one dedicated for the Pi (via USB-C), one for the 5V peripheral rail (Arduino, MCP2515 modules, LED strip). This isolates the Pi's power from noise generated by the LED strip and other devices.

| Part | Description | Approx. Price | Source |
|------|-------------|---------------|--------|
| **Pololu D36V6F5** | 5V 600mA step-down, tiny, efficient | $5 | Pololu |
| **Pololu D36V28F5** | 5V 2.8A step-down, automotive input (4.5-50V) | $10 | Pololu |
| **Pololu D36V50F5** | 5V 5A step-down (for heavier loads) | $15 | Pololu |
| **BINZET 12V to 5V 10A** | 50W industrial converter with screw terminals | $10 | Amazon |

**Wiring:**
```
Car 12V (ACC) ──[5A fuse]──┬──► Buck #1 (5V 3A) ──► Pi USB-C power
                            │
                            └──► Buck #2 (5V 2A) ──► Peripheral 5V rail
                                                      ├─► 3× MCP2515
                                                      ├─► Arduino Nano
                                                      ├─► WS2812B LEDs
                                                      └─► ESP32-S3 (if not USB-powered from Pi)
```

### Option C: UPS HAT with Automotive Input (Best Reliability)

Add a UPS (Uninterruptible Power Supply) HAT to the Pi. This provides battery backup for graceful shutdown when ignition is turned off, and clean power during engine cranking dips.

| Part | Description | Approx. Price | Source |
|------|-------------|---------------|--------|
| **Geekworm X728** | Pi UPS HAT, 18650 battery, 5V 6A | $30-40 | Amazon, Geekworm |
| **PiSugar S Plus** | Compact UPS for Pi, 5000mAh | $35-45 | Amazon |
| **Geekworm X1202** | Pi 5/4 UPS, USB-C PD in, auto shutdown | $25-35 | Amazon |
| **52Pi EP-0136** | UPS HAT with RTC, safe shutdown GPIO | $25-35 | Amazon, AliExpress |

**How it works:**
1. Car 12V → buck converter → UPS HAT input (charges battery)
2. UPS HAT → Pi (always clean 5V from battery)
3. When ignition off → 12V drops → UPS detects loss → signals Pi GPIO → Pi runs `shutdown -h now`
4. UPS keeps Pi alive for 10-30 seconds during shutdown
5. No more SD card corruption from hard power cuts

**Software side** — add a shutdown script:
```python
# /home/pi/MX5-Telemetry/pi/scripts/safe_shutdown.py
import RPi.GPIO as GPIO
import subprocess, time

SHUTDOWN_PIN = 4  # GPIO pin from UPS "power loss" signal

GPIO.setmode(GPIO.BCM)
GPIO.setup(SHUTDOWN_PIN, GPIO.IN, pull_up_down=GPIO.PUD_UP)

def shutdown_callback(channel):
    print("Power loss detected! Shutting down in 5 seconds...")
    time.sleep(5)
    subprocess.call(['sudo', 'shutdown', '-h', 'now'])

GPIO.add_event_detect(SHUTDOWN_PIN, GPIO.FALLING, callback=shutdown_callback, bouncetime=2000)
```

### Ignition-Switched Power Source

Rather than tapping OBD-II Pin 16 (which is always-on battery power on many vehicles), tap into a **switched 12V source** that turns off with the ignition:

| Source | Location (MX5 NC) | Notes |
|--------|-------------------|-------|
| **ACC fuse** | Interior fuse box, slot 15 (15A) | On with ACC/RUN, off when key removed |
| **Ignition fuse** | Interior fuse box, slot 20 (10A) | On with RUN only |
| **Radio ACC wire** | Behind head unit (Pioneer harness) | Already switched, easy access |
| **Add-a-fuse tap** | Any switched fuse slot | Cleanest installation |

**Recommended:** Use an **add-a-fuse** (piggyback fuse tap) on a switched fuse slot. This avoids splicing factory wiring.

| Part | Description | Approx. Price | Source |
|------|-------------|---------------|--------|
| **Nilight 12V Fuse Tap** (5-pack) | Mini blade add-a-fuse | $7-10 | Amazon (B077YCQ342) |
| **Chanzon Add-a-Fuse** (10-pack) | Mini/standard mix | $6-8 | Amazon |

### Recommendation

**Option C (UPS HAT) + ignition-switched power via add-a-fuse** is the most robust approach:
- Clean power to Pi at all times (battery buffered)
- Graceful shutdown prevents SD card corruption
- Survives engine cranking voltage dips
- Separate peripheral buck converter isolates LED/CAN noise from Pi

**Total cost:** ~$40-60 for UPS HAT + fuse tap + buck converter

## 2. SWC Volume Wire Splice for Pi Input

### The Problem

Currently, only **cruise control buttons** (RES+, SET-, ON/OFF, CANCEL) are readable via CAN bus ID `0x250` on MS-CAN. The audio/volume buttons (VOL+, VOL-, MODE, SEEK, MUTE) are **NOT transmitted on CAN** — they use a dedicated **analog resistance wire** between the steering wheel and the Axxess ASWC-1 module, which translates them into the Pioneer's wired remote protocol.

This means the telemetry system has only 4 buttons for navigation — not enough for all features (screen navigation, settings adjustment, brightness, volume, mode changes, etc.).

### How MX5 NC SWC Works

The MX5 NC steering wheel has two button clusters:

| Cluster | Buttons | Signal Type | CAN Available? |
|---------|---------|-------------|----------------|
| **Cruise Control** (right) | RES+, SET-, ON/OFF, CANCEL | Digital via CAN (0x250) | **Yes** — Pi reads these |
| **Audio** (left) | VOL+, VOL-, MODE, SEEK UP, SEEK DOWN, MUTE | **Analog resistance ladder** | **No** — dedicated wire |

The audio buttons work by placing different resistance values on a single wire (the SWC signal wire). Each button connects a different resistor to ground, and the receiving device (Axxess ASWC-1 or head unit) reads the voltage to determine which button is pressed.

### MX5 NC Audio SWC Resistance Values

| Button | Approximate Resistance | Voltage (at 3.3V ref) |
|--------|----------------------|----------------------|
| No press | Open circuit (∞) | 3.3V (pulled high) |
| VOL+ | ~87Ω | ~0.24V |
| VOL- | ~210Ω | ~0.55V |
| SEEK UP (>>I) | ~510Ω | ~1.17V |
| SEEK DOWN (I<<) | ~1kΩ | ~1.88V |
| MODE / SOURCE | ~2.2kΩ | ~2.49V |
| MUTE | ~4.7kΩ | ~2.89V |

> **Note:** These resistance values are approximate and specific to the MX5 NC. Measure your actual values with a multimeter before implementing. The voltage divider values above assume a 10kΩ pull-up to 3.3V.

### Implementation: ADC on Raspberry Pi

The Pi has no built-in ADC (analog-to-digital converter). There are two approaches:

#### Option A: External ADC Module (Recommended)

| Part | Description | Resolution | Approx. Price | Source |
|------|-------------|------------|---------------|--------|
| **ADS1115** | 16-bit I2C ADC, 4-channel, PGA | 16-bit | $3-8 | Amazon, Adafruit (#1085) |
| **ADS1015** | 12-bit I2C ADC, 4-channel, faster | 12-bit | $3-6 | Amazon, Adafruit |
| **MCP3008** | 10-bit SPI ADC, 8-channel | 10-bit | $3-5 | Amazon, Adafruit (#856) |

**Recommended: ADS1115** — 16-bit resolution easily distinguishes all resistance values, I2C doesn't conflict with existing SPI bus, and only needs 2 GPIO pins (SDA/SCL) which are currently unused.

#### Option B: Arduino ADC (Use Existing Arduino)

The Arduino Nano already has 6 analog inputs (A0-A5). Since it's already in the system and connected to the Pi via serial, you could:
1. Connect the SWC wire to Arduino analog pin A0
2. Arduino reads analog voltage, identifies button press
3. Arduino sends button event to Pi over existing serial connection

**Pros:** No new hardware needed
**Cons:** Arduino is physically located at the gauge cluster (may need a long wire run), and the serial link is already used for LED commands

### Wiring Diagram (ADS1115 Approach)

```
Steering Wheel SWC Signal Wire
        │
        ├──── To Axxess ASWC-1 (existing, keep connected)
        │
        └──── Splice ──┬──► ADS1115 Channel A0
                        │
                       10kΩ pull-up to 3.3V (if not already present)

ADS1115 Module:
  VCC ──► Pi 3.3V (Pin 1)
  GND ──► Pi GND (Pin 9 or any GND)
  SDA ──► Pi GPIO 2 / SDA1 (Pin 3)  ← Currently unused!
  SCL ──► Pi GPIO 3 / SCL1 (Pin 5)  ← Currently unused!
  A0  ──► SWC signal wire (spliced)
  ADDR ─► GND (I2C address 0x48)
```

### Software Implementation

```python
# pi/ui/src/swc_adc_handler.py
"""
Analog SWC Button Reader via ADS1115 ADC

Reads the resistance-ladder audio buttons from the steering wheel
by measuring voltage on the SWC signal wire through an ADS1115 ADC.
"""
import board
import busio
import adafruit_ads1x15.ads1115 as ADS
from adafruit_ads1x15.analog_in import AnalogIn
import time

# Voltage thresholds for each button (midpoints between adjacent buttons)
# Adjust these after measuring YOUR actual steering wheel resistances
BUTTON_THRESHOLDS = [
    (0.0,  0.40, "VOL_UP"),      # ~0.24V
    (0.40, 0.85, "VOL_DOWN"),    # ~0.55V
    (0.85, 1.50, "SEEK_UP"),     # ~1.17V
    (1.50, 2.20, "SEEK_DOWN"),   # ~1.88V
    (2.20, 2.70, "MODE"),        # ~2.49V
    (2.70, 3.10, "MUTE"),        # ~2.89V
    # Above 3.1V = no button pressed
]

class SWCADCReader:
    def __init__(self):
        i2c = busio.I2C(board.SCL, board.SDA)
        ads = ADS.ADS1115(i2c, address=0x48)
        ads.gain = 1  # ±4.096V range
        self.channel = AnalogIn(ads, ADS.P0)
        self._last_button = None
        self._debounce_time = 0.05  # 50ms debounce

    def read_button(self) -> str:
        """Read current button press. Returns button name or None."""
        voltage = self.channel.voltage
        for low, high, name in BUTTON_THRESHOLDS:
            if low <= voltage <= high:
                return name
        return None
```

### Important Wiring Notes

1. **Do NOT cut the existing SWC wire** — splice in parallel so the Axxess module still receives the signal
2. Use a **voltage divider** if needed to keep the signal within 0-3.3V (Pi GPIO tolerance)
3. The SWC wire is in the **steering column harness** — accessible behind the steering column covers or at the Axxess module connector
4. **Test with multimeter first** — verify actual resistance values for each button on YOUR car before setting thresholds
5. The Axxess ASWC-1 and the ADS1115 will both read the same wire — this is fine since ADC input impedance is very high (>10MΩ) and won't affect the resistance reading

### Where to Find the SWC Wire

On the MX5 NC, the SWC signal wire runs from the clock spring (steering column) to the Axxess ASWC-1 module (typically mounted behind the head unit or under the dash). The easiest splice point is:

1. **At the Axxess module connector** — the SWC input wire (usually blue/yellow or green)
2. **Behind the head unit** — where the ASWC-1 harness connects to the factory SWC harness
3. **At the steering column plug** — the multi-pin connector at the base of the column

## 3. Connector Upgrade: Replace DuPont with Automotive-Grade

### The Problem

DuPont (0.1" header pin) connectors are designed for prototyping, not for automotive environments. In a moving vehicle they suffer from:

- **Vibration loosening** — pins slowly walk out of sockets
- **Poor contact retention** — no locking mechanism
- **Corrosion** — exposed pins oxidize over time
- **Intermittent connections** — the #1 cause of random system glitches in-car

## 4. Additional Improvement Ideas

### 4A. Fused Power Distribution Block

Replace the breadboard + single fuse with a proper automotive mini fuse block:

| Part | Description | Approx. Price | Source |
|------|-------------|---------------|--------|
| **BlueSea ST Blade Fuse Block** (6-circuit) | Screw terminals, 30A bus | $25-35 | Amazon (B001P6FBOW) |

**Wiring:**
```
Car 12V (ACC switched) ──► Main 10A fuse ──► Fuse Block Bus
                                               ├─► 3A fuse → Buck #1 → Pi
                                               ├─► 2A fuse → Buck #2 → Peripherals
                                               ├─► 1A fuse → Future expansion
                                               └─► 1A fuse → Future expansion
```

Each circuit gets its own fuse. If the LED strip shorts, only that fuse blows — the Pi stays running.

### 4B. Strain Relief and Wire Management

Vibration doesn't just loosen connectors — it fatigues wires at solder joints and crimps. Add proper strain relief:

| Part | Description | Approx. Price | Source |
|------|-------------|---------------|--------|
| **Adhesive cable tie mounts** | Stick-on zip tie bases | $5/50pcs | Amazon |
| **Split wire loom** (1/4", 3/8") | Automotive wire conduit | $6-8/10ft | Amazon |
| **Kapton tape** | High-temp wire securing | $5/roll | Amazon |
| **Adhesive felt pads** | Anti-rattle for PCBs/modules | $3/pack | Amazon |
| **3M Dual Lock** | Reclosable fastener (stronger than velcro) | $8-12 | Amazon |

**Tips:**
- Secure every wire run with zip tie mounts every 6-8 inches
- Use split loom for any wire running more than 12 inches
- Add a drip loop before any connector (prevents strain on the connector)
- Pad all PCBs/modules with felt to prevent rattle against dash panels

### 4C. Ignition Sense Wire for Smart Power Management

Instead of a simple switched fuse tap, add a dedicated ignition sense wire to the Pi GPIO. This lets the Pi detect when the key is turned on/off and manage its own power state:

```
Ignition 12V ──[10kΩ + 20kΩ voltage divider]──► Pi GPIO 4 (3.3V safe)
                                                   │
                                                  10kΩ
                                                   │
                                                  GND
```

**Software behavior:**
- GPIO 4 HIGH → ignition on → Pi starts telemetry services
- GPIO 4 LOW → ignition off → Pi saves data, runs cleanup, shuts down gracefully
- Combined with UPS HAT, gives ~10-30 second shutdown window

### 4D. EMI/RFI Filtering

Automotive environments are electrically noisy (alternator whine, ignition sparks, relay switching). Add filtering to prevent random CAN errors or Pi glitches:
